Protocole de creation de personnage

Objectif : creer une fiche a partir des Markdown du repo, sans utiliser les images sources. Le protocole doit remplir toutes les cases calculables, identifier les choix manquants, distinguer donnees certaines / deduites / a arbitrer, et produire des recommandations sans masquer les choix du joueur ou du MD.

Chaque donnee importante est classee dans l'un de ces statuts. Vocabulaire rendu par .badge[data-status].

Les images ne doivent pas etre utilisees pendant la creation d'un personnage. Si une information manque dans les Markdown, le resultat indique que la base documentaire est incomplete.

Audit trail obligatoire

Chaque fiche doit permettre de comprendre pourquoi une case est remplie : pas seulement une valeur finale, mais la chaine de justification.

Format minimal recommande :

CaseValeurStatutPourquoi / preuve
ExempleExempleSource / CalculeFichier ou choix qui justifie la valeur.

Regles

  • toute valeur issue d'une classe doit citer la classe ou l'aptitude ;
  • toute valeur issue de l'espece doit citer le trait ou lignage ;
  • toute valeur issue de l'historique doit citer l'historique ;
  • toute valeur issue d'un don doit citer le don ;
  • tout calcul doit afficher la formule ;
  • tout doublon doit etre visible dans une section Conflits et alertes ;
  • un objet compose doit etre decompose sans perdre le lien, par exemple focaliseur druidique (baton de combat) produit a la fois un focaliseur et un objet baton de combat.
Besoin central : un joueur ou un MD doit pouvoir demander « pourquoi ce sort, cette maitrise ou cette arme est sur la fiche ? » et trouver la reponse sans refaire toute la creation. Voir le modele de graphe de provenance.

Ordre de traitement

  1. Creer ou ouvrir personnages/<nom>.md.
  2. Reporter les donnees fournies sans les modifier.
  3. Charger les donnees de classe depuis classes/<classe>.md.
  4. Charger l'historique et l'espece depuis origines.md.
  5. Charger les dons imposes ou choisis depuis dons.md.
  6. Charger l'equipement depuis equipement.md si les options sont choisies.
  7. Charger les sorts depuis sorts-selectionnes.md ou les listes de classe deja extraites.
  8. Calculer les valeurs derivees.
  9. Identifier les choix automatiques deja imposes par classe, espece, historique ou don.
  10. Identifier les choix restants.
  11. Identifier les conflits et doublons.
  12. Lister les choix manquants.
  13. Lister les incoherences.
  14. Proposer des recommandations separees des donnees certaines.

Choix automatiques, choix restants et conflits

Une fiche ne doit pas redemander un choix deja impose par une source :

  • une langue donnee automatiquement par une classe est notee Source, pas demandee ;
  • un sort donne automatiquement par une espece est note Source, pas compte comme choix libre ;
  • un don impose par un historique est ajoute automatiquement ;
  • un choix joueur qui duplique une option automatique est marque Conflit avec une remediation.

Quand une table permet de choisir ou de tirer au hasard : ne pas tirer sans accord explicite ; lister les options presentes dans les MD ; indiquer le de si la table existe ; sinon marquer Manquant documentaire.

Un conflit ne doit jamais etre corrige silencieusement. La fiche contient toujours les deux chemins concurrents et une remediation proposee.

Calculs systematiques

Caracteristiques

Noter la valeur, calculer le modificateur, verifier l'application des bonus d'historique, verifier qu'aucune valeur ne depasse 20 sauf regle explicite.

Niveau et maitrise

Niveau 1 : bonus de maitrise +2 ; sinon table de progression de creation-personnage.

Points de vie

PV niveau 1 = maximum du de de vie de classe + modificateur de Constitution. Niveaux suivants : moyenne fixe, jet de de, ou regle de table. Si les PV fournis ne correspondent pas, marquer Incoherent ou A arbitrer.

Classe d'armure

CA sans armure = 10 + modificateur de Dexterite. Verifier ensuite armure, bouclier, limite de Dexterite, aptitude, don, objet special. Si l'equipement n'est pas choisi, ne calculer que la CA sans armure et marquer la CA equipee manquante.

Initiative

Initiative = modificateur de Dexterite (+ dons, aptitudes, objets si documentes).

Jets de sauvegarde

JS non maitrise = modificateur de caracteristique ; JS maitrise = modificateur + bonus de maitrise. Les JS maitrises viennent principalement de la classe.

Competences

Non maitrisee = modificateur ; Maitrisee = modificateur + maitrise ; Expertise = modificateur + 2 x maitrise. Un doublon n'est pas cumule automatiquement : A arbitrer ou regle documentee.

Perception passive

Perception passive = 10 + modificateur de Sagesse (Perception) (+ maitrise si Perception maitrisee).

Sorts

DD des sorts = 8 + modificateur d'incantation + bonus de maitrise ; Attaque de sort = modificateur d'incantation + bonus de maitrise. Noter separement sorts mineurs, prepares, toujours prepares, emplacements, recuperation. Un sort donne par plusieurs sources est signale comme doublon possible / Conflit.

Controle qualite

Avant de considerer une fiche exploitable :

  • verifier que toutes les donnees viennent des Markdown ;
  • verifier les calculs principaux ;
  • verifier les doublons de maitrise, sorts ou equipement ;
  • verifier les incoherences entre PV, niveau, classe et Constitution ;
  • verifier que chaque choix manquant est liste ;
  • verifier que chaque recommandation est separee des donnees certaines.

Cas Medicis comme test

Medicis sert de test du protocole. Resultat attendu :

  • la fiche doit pouvoir etre initialisee depuis les Markdown ;
  • les calculs de base doivent etre remplis ;
  • les choix manquants doivent etre explicites ;
  • l'incoherence des PV doit etre visible ;
  • le doublon potentiel druidisme doit etre visible ;
  • aucune image ne doit etre consultee pendant la generation finale.
