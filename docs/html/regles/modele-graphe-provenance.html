Modele de graphe de provenance

Objectif : representer les regles, choix, calculs et cases de fiche comme un graphe oriente de provenance. Deux questions symetriques : depuis une case (« pourquoi ai-je cette valeur ? ») et depuis une source (« qu'est-ce que Guide / Druide / Elfe sylvestre m'apporte ? »).

Schema de demonstration

Exemple de graphe instancie sur Medicis. Noeuds types (Choix / Source / Effet / Case / Calcul) et aretes typees (choisit, accorde, calcule, remplit). Ce schema remplace les blocs ASCII des sources.

Choix joueur Source regle Effet / aptitude / sort Calcul Case de fiche

Aretes : choisit (personnage vers choix), accorde (source vers consequence automatique), calcule (donnees vers calcul et calcul vers case), remplit (effet vers case de fiche).

Deux graphes complementaires

Graphe de regles

Vient des Markdown de reference. Il decrit ce qu'une source accorde ou requiert, independamment d'un personnage.

Historique Guide → Don Initie a la magie (Druide) → 2 sorts mineurs + 1 sort niveau 1 + caracteristique d'incantation.

Classe Druide → Equipement option A → Focaliseur druidique (baton de combat) → objet + role focaliseur + arme.

Graphe de personnage

Instancie les regles avec les choix du joueur.

Medicis → Guide → Initie a la magie → choix manquants : 2 sorts mineurs, 1 sort niveau 1.

Medicis → Druide → Druidique → sort toujours prepare : communication avec les animaux.

Direction du graphe et DAG

Le graphe principal va des causes vers les consequences : source ou choix → aptitude / don / objet / maitrise / sort → calcul ou case de fiche. La lecture inverse (case ← effet ← source) est indispensable pour l'audit ; l'application doit afficher les deux.

Le graphe est principalement un DAG si on separe bien sources, choix, effets, cases et alertes. Il peut devenir cyclique si on melange une case finale avec sa propre justification.

A eviter : Sort prepare → section Sorts → validation des sorts → Sort prepare. Regle pratique : les calculs et validations lisent le graphe, mais ne redeviennent pas des prerequis de leurs propres sources.

Automatique, recommande, choisi

Ces trois statuts restent separes.

StatutEffet dans le grapheEffet dans l'interface
AutomatiqueAjoute directement un noeud de fiche.Rempli sans demander au joueur.
RecommandeAjoute une option conseillee.Pre-remplit ou met en avant, mais reste modifiable.
ChoisiValide une option.Devient une source effective pour la fiche.

Druide niveau 1 → recommande druidisme, recommande flammes, choix requis : 2 sorts mineurs. Si le joueur accepte, Medicis choisit druidisme + flammes. Le moteur detecte alors : druidisme deja accorde par Elfe sylvestre → Conflit : doublon.

Communication avec les animaux est un sort toujours prepare via Druidique. Il ne remplace pas les 4 sorts prepares recommandes du druide niveau 1, sauf regle contraire documentee.

Exemples de chemins

Pourquoi communication avec les animaux ?

communication avec les animaux ← sort toujours prepare ← Druidique ← Druide niveau 1 ← Classe Druide ← Medicis choisit Druide.

Qu'apporte Guide a Medicis ?

Medicis choisit Guide → Historique Guide → Discretion, Survie, Outils de cartographe, Initie a la magie (Druide), Equipement Guide A. Puis Initie a la magie → 2 sorts mineurs + 1 sort niveau 1 + caracteristique d'incantation.

Pourquoi un baton de combat ?

Baton de combat ← objet de depart ← Focaliseur druidique (baton de combat) ← Equipement Druide option A ← Classe Druide ← Medicis choisit Druide.

Baton de combat → role focaliseur druidique → case Incantation ; et baton de combat → role arme courante → attaque +2, 1d6 contondants ou 1d8 a deux mains → case Armes et attaques.

Pour les fiches humaines, les tables « Depuis les cases » et « Depuis les sources » restent plus lisibles. Pour une application, le format noeuds / aretes est plus robuste. Le modele accepte les relations many-to-one, one-to-many et many-to-many.

Controle qualite du graphe

Un graphe de personnage est acceptable si :

  • chaque case importante a au moins un chemin de provenance ;
  • chaque choix automatique est relie a sa source ;
  • chaque choix requis est visible comme noeud Choix requis ;
  • chaque conflit a les deux chemins concurrents ;
  • chaque calcul garde ses entrees ;
  • aucun effet n'est seulement « devine » sans statut ;
  • les objets composes sont decomposes en objet concret et roles.
